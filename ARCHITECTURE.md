# Project Architecture

This is a TanStack Start app with Electric SQL for real-time sync, generated by Electric Studio.

## Stack

- **Framework**: TanStack Start (React, file-based routing, SSR)
- **Sync**: Electric SQL shapes → TanStack DB collections → `useLiveQuery`
- **Database**: Postgres (Drizzle ORM for schema + migrations)
- **Validation**: Zod v4 (derived from Drizzle schema via `drizzle-zod`)
- **Styling**: Tailwind CSS + Radix Themes
- **Dev Server**: Vite
- **Testing**: Vitest

**Intent skills:** This template runs `tanstack-intent install` on `postinstall` to populate `.claude/skills/` with per-library SKILL.md docs from `node_modules` (Electric SQL client, TanStack DB, Drizzle, etc.). These skills are read by Claude/Copilot/etc. at runtime and stay in sync with library version updates.

## Directory Structure

```
src/
  routes/             File-based routes (TanStack Router)
    __root.tsx        Root layout — NEVER set ssr:false here
    index.tsx         Home page
    api/              Server-side API routes (handlers, not components)
  components/         Shared React components
    ClientOnly.tsx    SSR wrapper — children must be a render function: {() => <Foo />}
    Toaster.tsx       Toast notification provider + useToast() hook (wraps @radix-ui/react-toast)
  db/
    schema.ts         Drizzle pgTable definitions (source of truth for types)
    index.ts          Drizzle client setup (reads DATABASE_URL from env)
    zod-schemas.ts    Zod validators derived from schema via drizzle-zod
    utils.ts          parseDates helper + generateTxId
    collections/      TanStack DB collection definitions (one file per entity)
  lib/
    client-url.ts     absoluteApiUrl() — resolves API URLs for SSR and client
    electric-proxy.ts Electric shape proxy helper for server routes
tests/
  helpers/
    schema-test-utils.ts  generateValidRow / generateRowWithout for testing
scripts/
  preflight.mjs       Build-time checks (SSR safety, useLiveQuery patterns, etc.)
```

## Key Patterns

### Data Flow
1. Drizzle schema (`src/db/schema.ts`) defines Postgres tables
2. `drizzle-kit generate && drizzle-kit migrate` creates/updates tables
3. Electric shape proxy (`src/routes/api/<entity>.ts`) syncs table → client
4. TanStack DB collection (`src/db/collections/<entity>.ts`) subscribes to the shape
5. `useLiveQuery()` in components reactively queries the collection
6. Mutations: `collection.insert/update/delete` → API route → Postgres → syncs back

### Query Rules (CRITICAL)
- `useLiveQuery()` returns `{ data }`, NOT the array. Always: `const { data: items = [] } = useLiveQuery(...)`
- `.where()` MUST use `eq()`, `gt()`, etc. — NEVER use `===`, `!==`, `<`, `>`
  - WRONG: `.where(({ todo }) => todo.completed === true)`
  - RIGHT: `.where(({ todo }) => eq(todo.completed, true))`
- Import operators from `@tanstack/db`: `import { eq, gt, and, or, not } from "@tanstack/db"`

### SSR Rules
- Leaf routes using `useLiveQuery`: set `ssr: false` on route options
- `__root.tsx`: NEVER set `ssr: false` — use `<ClientOnly>{() => <Component />}</ClientOnly>`
- `<ClientOnly>` children MUST be a render function, not JSX

### HTML Rules
- NEVER nest `<button>`, `<a>`, or `<input>` inside another `<button>` or `<div role="button">` — causes hydration errors and broken click handling
- For list items with both a row click and a delete button: use a `<div>` row with `onClick` (no `role="button"`) and a separate `<button>` inside with `e.stopPropagation()`

### API Routes (Server-Side)
- File: `src/routes/api/<name>.ts`
- Export `Route` with `server.handlers.GET/POST/PUT/DELETE`
- Electric shape proxy: forward requests to Electric Cloud with secret injection
- NEVER expose secrets to the client — all cloud calls go through server routes

### Dev Server
- `pnpm dev` starts Vite on port 5174
- Bring your own Postgres + Electric SQL — see `.env.example` for the required variables

### Environment Variables
- `DATABASE_URL` — Postgres connection string
- `ELECTRIC_SOURCE_ID` — Electric Cloud source ID
- `ELECTRIC_SECRET` — Electric Cloud auth secret
- Written to `.env` by the credential provisioning flow, read by server routes
